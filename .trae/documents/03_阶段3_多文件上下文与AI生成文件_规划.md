## 2. 下一阶段目标（Goals）

### G1：多文件上下文（AI 可读取多个文件）

- 用户可选择多个文件作为上下文，让 AI 在回答/改码时“看到这些文件内容”。
- 上下文可控：避免一次注入过多内容导致 token 爆炸。

### G2：AI 生成/修改多个文件（即使没有 Function Call）

- 支持“在没有现成目标文件的情况下，让 AI 生成新文件并写入内容”。
- 支持“一次请求修改多个文件”（例如：新增 `index.html`、`styles.css`、`app.js`）。
- 所有文件变更都必须经过用户确认（Diff / Review），再写回工作区 store。

### G3：面向后续 Function Call 的兼容设计（Optional）

- 本阶段不强制上 Function Call，但输出协议与内部接口要为未来切换“tool calls”做好准备（例如统一的 `FilePatch` 结构）。

## 3. 非目标（Non-goals）

- 不做真实本地文件系统读写（保持模拟工作区）。
- 不做 AI 补全 / Inline Edit（选区局部替换）除非作为 G2 的附带能力被自然覆盖。
- 不做 Godot 运行/调试/与引擎通信。

## 4. 用户体验（UX）与交互流程

### 4.1 多文件上下文选择

**入口：**

- 在文件树每个文件旁增加“加入上下文”勾选/按钮（或在文件树顶部提供“上下文管理器”）。

**行为：**

1. 用户勾选若干文件（例如 `/web/index.html`、`/web/styles.css`）。
2. Chat 发送时，系统把这些文件内容拼装成上下文块，注入到 system prompt。
3. AI 回答时可以引用这些文件并给出跨文件修改建议。

**上下文策略（建议默认）**

- “完整内容模式”：对体积小的文件直接全量注入。
- “截断模式”：对大文件按最大字符数截断（例如每文件最多 12k chars），并保留头尾片段。
- “摘要模式（后续）”：先生成/维护每文件摘要，优先注入摘要+关键片段（可作为未来增强）。

### 4.2 AI 生成/修改文件（File Patch 模式）

**核心：让 AI 输出一个可解析的“文件变更包”，前端解析并应用。**

**交互：**

1. 用户在 Chat 输入需求：例如“帮我生成一个三文件的静态页面：index.html + styles.css + app.js”。
2. AI 必须输出一个结构化块（见 5.2 输出协议）。
3. 前端解析到 `FilePatch[]`：
   - 若文件不存在：创建文件节点 + 写入内容
   - 若文件存在：走 Diff 审核后更新
4. 弹出“变更总览（multi-file review）”：
   - 列出新增/修改的文件列表
   - 每个文件可展开 diff
   - 用户点击“全部应用”或逐个应用

**失败兜底：**

- AI 输出不符合协议：提示“未检测到可应用的文件变更包”，保留普通对话展示，不修改任何文件。

## 5. 数据结构与协议

### 5.1 工作区数据结构（扩展）

在现有 [workspaceStore.ts](file:///Users/goumomo/Desktop/Project/ai-ide/ai-ide-demo/src/store/workspaceStore.ts) 基础上扩展：

**WorkspaceStore（新增字段/动作）**

- `selectedContextPaths: string[]`
- `toggleContextPath(path: string)`
- `clearContext()`
- `applyFilePatches(patches: FilePatch[])`（先做校验，再写入 store）

### 5.2 FilePatch 输出协议（无 Function Call）

AI 必须在回答中包含一个代码块，语言标记为 `json`，内容是：

```json
{
  "type": "ai-ide-file-patch",
  "version": 1,
  "patches": [
    {
      "path": "/web/index.html",
      "action": "upsert",
      "content": "<!DOCTYPE html>..."
    },
    {
      "path": "/web/styles.css",
      "action": "upsert",
      "content": "body { ... }"
    }
  ]
}
```

**字段约束**

- `type`：固定为 `ai-ide-file-patch`（用于可靠识别）
- `version`：协议版本（便于演进）
- `patches[].path`：必须是工作区内的绝对路径（以 `/` 开头）
- `patches[].action`：本阶段只需要 `upsert`（未来可扩展 `delete/rename`）
- `patches[].content`：完整文件内容（本阶段不做增量 patch）

### 5.3 多文件 Review（Diff）数据结构

沿用现有单文件 Diff 逻辑，但增加聚合层：

- `MultiFileChange`：
  - `path: string`
  - `originalContent: string`（不存在则为空）
  - `modifiedContent: string`
  - `changeType: 'added' | 'modified'`

UI 层新增 `MultiDiffModal`：

- 列表视图（文件维度）
- 详情视图（复用或内嵌 diff 行渲染）

## 7. 验收标准（Acceptance Criteria）

### A. 多文件上下文

- A1：用户可在文件树勾选至少 1 个文件作为上下文；UI 有明确可见状态。
- A2：发送消息时，AI 能正确引用所选文件内容（可用人工验证：询问某个文件里是否包含某段字符串）。
- A3：上下文注入有上限保护：选择超多文件/超大文件不会导致页面卡死或请求体过大（至少有截断/提示）。

### B. AI 生成/修改多文件（无 Function Call）

- B1：当用户提出“生成多个文件”需求时，AI 能返回 `ai-ide-file-patch` JSON 块。
- B2：前端能解析该 JSON，展示多文件变更审核弹窗（至少显示 2 个文件的新增/修改）。
- B3：用户点击“全部应用”后：
  - 新文件出现在文件树中
  - 文件内容写入成功
  - 若其中某文件是 HTML，切换到该文件可预览
- B4：当 AI 不输出协议块或输出非法 JSON：
  - 不修改任何文件
  - 页面不报错、不崩溃

## 8. 验证方式（Verification）

- 手动验证：
  - 选择两个文件作为上下文，问 AI “styles.css 里有哪些 class？”能回答并引用内容
  - 让 AI 生成 3 个文件（html/css/js），出现多文件审核弹窗并成功应用
  - 测试非法 path（让 AI 输出 `../x`）应被拒绝
- 构建验证：
  - `npm -C ai-ide-demo run build` 通过

## 9. 未来扩展：Function Call 接入（备选路线）

当未来切换到支持 tools 的模型/网关时：

- 将 `applyFilePatches` 的能力映射为 tool：`create_file/update_file/open_file/list_files/read_file`
- Chat 端不再依赖“模型输出 JSON 协议块”，而是由模型调用 tool，前端展示同样的 multi-file review
- 现有 `FilePatch` 协议仍保留作为回退通道（兼容不支持 tools 的模型）
